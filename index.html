It works!

This is the default web page for the VSCP daemon. This is a fully working web server so you can add HTML files to the folder you have setup using <webrootpath> in the VSCP configuration file and serve as from any other webserver.

Note that /vscp is a reserved path and you should not us it in your own folder structure as it will take you to this reserved page.

Setting things up

To test the websocket interface of the VSCP daemon you first edit the settings.js file which after installation is located in the js subfolder of the web root filesystem which normally is at /srv/vscp/web on a linux system and at C:\Programdata\VSCP\web on a windows system.

The default content in this file is: 

    var settings = {

        user: "admin",
        authdomain: "mydomain.com",
        passwordHash: "d50c3180375c27927c22e42a379c3f67", // Hash on "user:authdomain:password"
        url: "ws://192.168.1.8:8080" // Non SSL (to be able to use from remote machine)
        
    };
which is the default admin credentials for

  • Username: admin
  • Password: secret
  • Domain: mydomain.com

needless to say you should change login settings as soon as you use your system more seriously.

The last part in the settings.js file to change is the url for your server. The default for the websocket samples is to use a non-ssl connection (unsecure). If you want to use ssl you should use "wss://" as prefix instead of "ws://"

The important first test

When you have edited and saved this information you should verify that the websocket login test script works. Nothing else will work if this part does not work. Follow the output messages and if the connection is established successful, everything is fine, otherwise something is bad with your login informations.
